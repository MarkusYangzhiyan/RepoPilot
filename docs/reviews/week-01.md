# RepoPilot 第 1 周复盘

> 记录日期：2026-07-06
> 所属里程碑：M0 产品冻结与基准
> 覆盖任务：Day 1–Day 5

## 1. 本周结论

第 1 周完成了产品定义、V1.0 范围冻结、候选仓库可行性调研和 Click 基线复现。项目目前已经回答三个基础问题：RepoPilot 要解决什么问题、V1.0 明确不做什么、首个目标仓库能否在固定环境中重复运行测试。

本周没有开始业务代码开发。这是有意的范围控制：在产品边界、评测对象和测试基线尚未明确前，不提前搭建可能返工的平台代码。

除本复盘与 ADR-001 尚待人工验收和合并外，Day 1–Day 4 的产物均已通过 Pull Request 进入 `main`。

## 2. 实际完成项

### Day 1：产品章程

- 建立 `docs/product/charter.md`。
- 明确核心用户、问题、输入、输出和非目标。
- 形成 1–2 分钟产品阐述。
- 将产品定位为“有证据、有测试、有审查的候选补丁”生成平台，而不是万能 Agent。

### Day 2：V1.0 范围

- 建立 `docs/product/scope-v1.md`。
- 冻结 Python 3.12、pytest、单仓库、单 Issue、单租户边界。
- 明确单任务默认限制：最多 5 个文件、500 行总变更、3 轮 Patch–Test–Review、30 分钟。
- 明确两个 HITL 节点、沙箱边界和 Draft PR 之前的人工确认要求。

### Day 3：候选仓库调研

- 建立 `docs/research/repository-candidates.md`。
- 核验 Click、Typer、Starlette、HTTPX 的许可证、默认分支、Python 版本、测试入口和贡献规则。
- 确认数据集角色：Click 为主开发集、Typer 为相近领域验证集、Starlette 为跨领域盲测集、HTTPX 为非阻塞高难度扩展集。
- 识别 Typer–Click 和 Starlette–HTTPX 的跨仓库答案泄漏风险。

### Day 4：Click 可复现基线

- 在 RepoPilot 仓库外克隆 Click，避免将第三方源码提交到本项目。
- 锁定 Click tag `8.4.2` 和 commit `b2e30a175449cfda909ee4fbf4a29a6a071cad53`。
- 使用 Python 3.12.3、uv、tox 和项目锁文件建立测试环境。
- 连续两次运行官方默认 Python 3.12 测试，结果数量完全一致。
- 建立 `docs/research/click-baseline.md`，记录环境、命令、结果、耗时和限制。

## 4. 本周已经冻结的决策

- V1.0 只支持 Python 3.12 和以 pytest 为主的项目。
- 每个任务只绑定一个仓库、一个 Issue 和一个固定 base commit。
- 采用单租户内部部署，不建设多租户商业 SaaS。
- 使用 LangGraph 管理状态、条件路由、Checkpoint 和 HITL。
- 使用可评测的混合代码 RAG，不以单一向量检索替代代码证据。
- 修复计划和候选补丁必须分别经过人工审批。
- 不自动合并 PR，不自动部署。
- Click 作为首个开发基线，Day 4 不提前扩展到其他仓库。

## 5. 本周调整与偏差

### 修改规模从 300 行调整为 500 行

范围评审时认为 300 行可能过早排除合理的小型修复，因此将默认上限调整为 500 行。统计口径明确为新增行数与删除行数之和，避免通过同时增删代码规避限制。该调整已同步到 V1 Plan、范围文档和项目状态。

### Click 基线从 8.1.8 改为 8.4.2

执行前曾考虑使用 8.1.8，但当时尚未绑定具体历史 Issue，选择较旧版本缺少充分依据。最终改用当时最新稳定 tag 8.4.2 建立发布基线；未来复现具体 Issue 时再切换到修复前 base commit。

### Git 克隆出现 TLS 中断

首次完整克隆出现 `GnuTLS recv error` 和 `early EOF`。由于目录没有残留不完整仓库，随后使用 HTTP/1.1、`--depth 1` 和 `--single-branch` 完成固定 tag 的浅克隆。该方式满足 Day 4，但不满足后续历史 Issue 全量检索，需要按任务补充对应历史 commit 或完整历史。

## 7. 当前主要风险

| 风险 | 影响 | 当前控制措施 | 后续验证 |
| --- | --- | --- | --- |
| 历史 PR 或跨仓库代码泄漏答案 | 评测结果失真 | 固定 base commit、时间过滤、隔离真实 PR Diff | 建立数据 Schema 和泄漏检查 |
| Typer 内嵌 Click 代码 | 开发集污染验证集 | 仓库与时间双重切分 | 采集任务时检查相似代码 |
| Starlette 与 HTTPX 历史交叉 | 盲测泄漏 | 跨仓库时间过滤 | 盲测冻结前审计关联变更 |
| CLI 和异步问题受平台影响 | 复现不稳定 | 固定操作系统、Python、依赖和命令 | 增加跨平台任务标签 |
| 30 周范围仍然较大 | 延期 | Python-only、单仓库、单租户 | 每周检查范围变更和实际产物 |
| 沙箱执行不可信仓库代码 | 宿主机安全风险 | 设计要求默认断网、无密钥、无 Docker Socket | M4 前完成威胁模型和安全测试 |

## 8. 第 2 周决策与方向

第 2 周仍属于 M0，不进入平台功能扩张。开始前需要先补充对应的每日任务，并围绕以下结果安排工作：

1. 定义历史 Issue 任务记录 Schema、准入条件和防泄漏检查。
2. 从 Click 等既定仓库中筛选第一批可复现历史 Issue，目标与 M0 的 10 条任务要求对齐。
3. 为单 Agent + 文本搜索基线明确最小输入、输出、工具和评价指标。
4. 保持 Python、pytest、单仓库、单 Issue 和单租户边界，不提前实现多语言或多租户能力。
