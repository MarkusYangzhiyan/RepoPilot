# ADR-001：V1.0 先采用 Python 单租户方案

> 状态：Accepted
> 决策日期：2026-07-06
> 决策范围：RepoPilot V1.0

## 1. 背景

RepoPilot 的目标是将单个 GitHub Issue 转换为有代码证据、有测试验证、有独立审查的候选补丁。V1.0 需要同时证明代码检索、根因调查、LangGraph 状态编排、人工审批、沙箱执行、测试闭环、权限隔离、审计和恢复能力。

如果从第一版同时支持多种语言和多租户 SaaS，项目必须额外处理不同语言的解析器、构建系统、测试框架、依赖管理、沙箱镜像、租户隔离、配额、计费和数据生命周期。上述工作会显著扩大实现与评测矩阵，并削弱对核心软件维护闭环的验证深度。

当前项目周期为 30 周，优先目标是完成一个可复现、可评测、可审计的企业内部部署版本，而不是尽早扩大语言数量和商业形态。

## 2. 决策

RepoPilot V1.0 采用以下边界：

- 目标代码语言限定为 Python 3.12。
- 主要测试体系限定为 pytest。
- 每个任务只处理一个 GitHub 仓库中的一个 Issue。
- 部署形态限定为单租户内部部署，使用 Linux 单机或单节点 Docker Compose。
- 单个租户内仍支持 Admin、Maintainer、Viewer 三种角色，以及仓库级授权、审计、删除和恢复能力。
- V1.0 不实现多语言、多租户计费、跨仓库原子修改、自动合并和自动部署。

“单租户”不等于“单用户”或“不需要权限”。它表示一个部署实例服务一个组织或团队，但该组织内部仍然有多个用户和角色。

## 3. 决策理由

### 3.1 优先验证深度

Python 具有标准 AST、成熟的类型与测试工具链，并且 Click、Typer、Starlette、HTTPX 提供了从 CLI 到异步 Web/HTTP 的难度梯度。限制语言后，可以把开发时间用于混合代码 RAG、证据引用、测试闭环和失败分析，而不是重复适配多种工具链。

### 3.2 保持评测可比

统一 Python 与 pytest 后，历史 Issue 可以使用相近的数据结构、测试结果和难度标签。单 Agent、Dense RAG、Hybrid RAG 和多 Agent 方案之间的差异更容易归因，不会被语言和构建系统差异淹没。

### 3.3 降低不可信代码执行风险

只支持一种主要语言和测试体系，可以先建立明确的 Docker 镜像、命令白名单、依赖安装阶段和资源限制。多语言会扩大编译器、包管理器、网络访问和原生依赖的攻击面。

### 3.4 单租户仍能体现企业工程能力

单租户部署不需要计费和跨租户资源调度，但仍然要求 GitHub 授权、RBAC、仓库权限、凭据保护、审计、任务隔离、删除、备份和恢复。它能够证明企业内部部署能力，同时控制商业 SaaS 带来的额外范围。

### 3.5 符合当前工期和交付目标

项目需要在 30 周内交付可运行系统、真实历史 Issue 评测、消融实验、安全演练和部署文档。Python 单租户方案让工期集中在可验证成果上，并为未来扩展保留真实数据依据。

## 4. 考虑过的方案

### 方案 A：多语言、多租户 SaaS

同时支持 Python、Java、Go 等语言，并从第一版提供跨组织多租户服务。

未选择原因：语言适配、租户隔离、配额、计费、合规和运维矩阵同时扩大，无法在当前周期内保证软件维护闭环的深度和安全性。

### 方案 B：Python 多租户 SaaS

先限制为 Python，但从第一版实现商业多租户能力。

未选择原因：虽然语言范围可控，但仍需要租户级密钥、数据隔离、资源调度、用量统计、计费和更复杂的删除流程。这些工作不会直接证明 Issue 根因调查和补丁验证能力。

### 方案 C：Python 单用户本地脚本

只做命令行脚本，不实现身份、权限、队列、审计、恢复和内部部署能力。

未选择原因：实现成本最低，但无法满足企业内部部署目标，也不能证明长任务恢复、权限边界和可审计性。

### 方案 D：Python 单租户内部部署

保留企业需要的多用户角色、授权、审计、隔离和恢复，同时不承担多租户商业 SaaS 的复杂度。

选择该方案。

## 5. 正向后果

- 可以集中完成 Python AST、符号关系、混合检索和 Reranker 的可量化评测。
- 可以先形成统一的 pytest 复现、目标测试和回归测试契约。
- Docker 沙箱和命令策略的初始攻击面更小。
- 单租户数据库、对象存储、队列和备份方案更容易部署和演练。
- 开发集、验证集和盲测集可以形成明确的仓库与难度梯度。
- 项目更可能在既定周期内交付完整闭环，而不是留下多个浅层适配器。

## 6. 负向后果

- V1.0 无法处理 Java、Go、Rust、JavaScript 等仓库。
- 以非 pytest 为主要测试体系的 Python 项目不属于承诺范围。
- 一个任务不能同时修改多个仓库或解决多个独立 Issue。
- 不能用 V1.0 证明商业多租户计费和大规模租户调度能力。
- 后续增加语言时，需要重新设计解析、索引、测试命令、沙箱镜像和评测数据。
- 单租户可能降低早期产品覆盖面，但这是为了换取交付深度和可信证据。

## 7. 实施约束

- 任何节点不得静默扩大到非 Python 或非 pytest 主测试体系。
- 任务创建时必须绑定一个租户、一个仓库、一个 Issue 和一个固定 base commit。
- 每个任务必须执行仓库授权检查、计划审批和交付审批。
- 不得为了提高自治程度而弱化沙箱、权限、审计、评测或恢复能力。
- Draft PR 必须经过显式人工确认，Merge 和 Deploy 始终由人类完成。
- 新语言或多租户能力必须通过新的 ADR 和范围评审，不能作为普通功能顺手加入。

## 8. 重新评估条件

只有在以下条件形成真实证据后，才考虑扩展语言或租户模型：

- Python 单仓库闭环已经稳定通过开发集、验证集和盲测集。
- 两个 HITL 节点、Checkpoint 恢复、权限隔离和沙箱安全已经达到 V1.0 门禁。
- 至少五名非作者用户完成试用并留下结构化反馈。
- 数据显示非 Python 或多租户需求具有足够优先级，且收益高于新增实现与评测成本。
- 已有明确的新增语言测试仓库、历史 Issue 数据、沙箱策略和维护预算。

重新评估不代表自动改变本 ADR。任何调整都必须先更新产品章程、范围文档和 V1 Plan，并创建后续 ADR。
