# RepoPilot 产品章程

> 文档状态：V1.0 草案
> 更新日期：2026-07-06

## 一句话定义

RepoPilot 是一个把 GitHub Issue 转换为“有代码证据、有测试验证、有独立审查的候选补丁”的软件维护 Agent。

## 1–2 分钟产品阐述

RepoPilot 面向维护 Python 项目的个人开发者和研发团队。它解决的问题不是“怎样让大模型多写代码”，而是接手陌生仓库或处理 GitHub Issue 时，怎样更快找到可信根因，并通过测试确认修改解决了原问题，同时没有破坏原本正常的功能。

用户提供 GitHub 仓库、目标 commit 和 Issue。系统先固定代码版本，提取期望行为与复现条件，再结合关键词检索、向量检索和代码结构分析，找到相关文件与符号，形成带路径、行号和 commit 的证据。职责独立的 Agent 随后完成根因调查、修复计划、补丁生成、测试与代码审查。

它不会拿到 Issue 就直接修改代码。修复计划必须先由人批准；修改只在临时 worktree 和隔离的 Docker 沙箱中执行；补丁还必须通过目标测试、回归测试和独立 Review Agent 检查。用户再次确认后，系统才会导出 Diff、报告或创建 Draft PR。

RepoPilot 的价值是让结论有证据、修改有测试、关键动作可审批，过程可审计、可恢复。V1.0 聚焦 Python、pytest、单仓库、单个 Issue 和单租户内部部署，不自动合并 PR，不自动部署，也不做“万能 Agent”。

## 核心用户

- 维护 Python 开源项目的个人开发者或小团队。
- 刚接手陌生代码库、需要快速定位问题的开发者。
- 希望在内部仓库中引入可控 Agent 工作流的研发团队。

## 要解决的问题

- 阅读陌生代码和定位 Issue 根因耗时，并且结论容易缺少代码证据。
- 大模型可能跳过复现与验证，直接生成看似合理但不可证明的补丁。
- 自动修改代码存在无关 Diff、回归、越权和不可审计风险。
- Agent 任务执行时间长，中断后需要能够恢复，而不是从头重跑。

## 输入

- 一个已授权的 GitHub Python 仓库。
- 一个固定的目标 commit。
- 一个 GitHub Issue，以及必要的补充约束和验收条件。

## 输出

- 带文件路径、行号和 commit 的证据与根因说明。
- 可供人工审批的修复方案与测试计划。
- 在隔离环境中生成并验证的候选补丁。
- 测试、Lint、类型检查和独立审查结果。
- 可审计、可复现的任务报告，以及经人工确认后创建的 Draft PR。

## V1.0 非目标

- 不自动合并 PR，不自动部署生产环境。
- 不处理跨仓库原子修改或任意大型功能开发。
- 不保证解决所有 Issue，也不把模型输出当作事实。
- 不支持 Python 和 pytest 之外的语言与主要测试体系。
- 不建设多租户商业 SaaS、计费系统、IDE 插件或移动端。
- 不以牺牲沙箱、权限、审计、评测和恢复能力换取更高自治程度。
