# RepoPilot V1.0 范围说明

> 文档状态：V1.0 草案
> 更新日期：2026-07-06
> 适用里程碑：M0–M7

## 1. 文档目的

本文档定义 RepoPilot V1.0 可以处理什么、明确不处理什么，以及单个任务必须遵守的规模和执行限制。它用于产品验收、任务准入、Agent 路由和后续评测，不用于描述 V1.1 及更远版本的设想。

## 2. 范围总述

RepoPilot V1.0 是一个单租户、企业内部可部署的 GitHub Issue 软件维护 Agent。它针对单个 GitHub Python 仓库中的单个 Issue，在固定 commit 上完成证据检索、根因调查、修复计划、人工审批、隔离修改、测试、独立审查和候选补丁交付。

V1.0 的目标不是自主完成任意软件开发，而是在受控范围内产出“有证据、有测试、有审查”的候选补丁。

## 3. 基本任务契约

每个 RepoPilot 任务必须同时满足以下条件：

- 绑定一个租户。
- 绑定一个已授权的 GitHub 仓库。
- 绑定一个固定且可解析的 base commit。
- 绑定一个 GitHub Issue。
- 只修改该仓库的一个临时 worktree。
- 在修改前经过修复计划人工审批。
- 在交付前经过候选补丁人工审批。
- 最多交付 Diff、任务报告或 Draft PR，不自动合并和部署。

## 4. 支持矩阵

| 维度 | V1.0 支持范围 | 限制与说明 |
| --- | --- | --- |
| 部署模式 | 单租户内部部署 | Linux 单机或单节点 Docker Compose |
| 用户角色 | Admin、Maintainer、Viewer | 权限由 RBAC 控制 |
| 代码托管 | GitHub 公共仓库、已授权的 GitHub 私有仓库 | 通过 GitHub App 或 OAuth 使用最小权限访问 |
| 仓库数量 | 每个任务一个仓库 | 不执行跨仓库修改或跨仓库事务 |
| Issue 数量 | 每个任务一个 Issue | 多个问题必须拆分为独立任务 |
| 代码语言 | Python 3.12 | 目标仓库必须能够在受控的 Python 3.12 环境中安装并运行 |
| 测试体系 | pytest | 可运行仓库已配置且策略允许的补充测试命令 |
| 任务类型 | Bug、兼容性问题、小范围重构、明确的测试修复 | 必须具有可描述的期望行为和验收条件 |
| 代码版本 | 固定 base commit | 所有证据、索引、修改和报告均绑定该 commit |
| 代码检索 | 关键词、向量、符号和调用关系的混合检索 | 证据必须包含路径、行号和 commit |
| 修改环境 | 临时 Git worktree 与 Docker 沙箱 | 默认断网、限制资源、不挂载宿主凭据或 Docker Socket |
| Agent 流程 | Issue 分析、代码调查、修复规划、补丁、独立审查 | Agent 之间使用结构化 Schema 传递信息 |
| 人工控制 | 计划审批、交付审批 | 两个 HITL 节点都必须可中断和恢复 |
| 交付结果 | Diff、测试与审查报告、经确认的 Draft PR | 不自动 Merge，不自动 Deploy |

## 5. 不支持矩阵

| 维度 | V1.0 不支持 | 处理方式 |
| --- | --- | --- |
| 部署模式 | 多租户商业 SaaS、Kubernetes 高可用、多地域容灾 | 记录为后续版本候选，不在 V1.0 实现 |
| 代码托管 | GitHub 之外的 Git 服务、本地未授权仓库 | 拒绝创建任务 |
| 仓库范围 | 跨仓库原子修改、一个任务修改多个仓库 | 要求拆分或人工处理 |
| Issue 范围 | 一个任务同时解决多个独立 Issue | 要求拆分为独立任务 |
| 语言范围 | Java、Go、Rust、JavaScript 等非 Python 项目 | 拒绝自动执行 |
| 测试范围 | 以非 pytest 体系为主要验收方式的项目 | V1.0 不承诺支持 |
| 任务规模 | 大型功能、架构重写、大规模迁移、无明确边界的重构 | 拒绝或要求人工拆分 |
| 自动化权限 | 自动合并 PR、自动部署、绕过人工审批 | 策略层禁止 |
| 执行权限 | 任意宿主命令、运行时自由联网、读取宿主密钥 | 沙箱与工具策略拒绝 |
| 客户端 | IDE 插件、原生桌面端、移动端 | 不进入 V1.0 计划 |
| 商业能力 | 计费、套餐、跨租户数据共享 | 不进入 V1.0 计划 |
| 成功承诺 | 保证解决所有 Issue | 允许以证据不足、无法复现或超出范围结束任务 |

## 6. 单任务规模与预算限制

| 限制项 | V1.0 默认值 | 达到限制时的行为 |
| --- | ---: | --- |
| 仓库数量 | 1 个 | 拒绝跨仓库执行 |
| Issue 数量 | 1 个 | 要求拆分任务 |
| 修改文件数量 | 不超过 5 个 | 暂停并请求人工缩小范围或终止 |
| Diff 总变更行数 | 不超过 500 行 | 暂停并请求人工缩小范围或终止 |
| Patch–Test–Review 修正次数 | 最多 3 轮 | 停止自动修正并交由人工决定 |
| 单任务执行时间 | 默认不超过 30 分钟 | 保存 Checkpoint 后暂停 |
| 人工审批节点 | 2 个 | 未批准不得进入下一受控阶段 |
| Token、成本和工具调用 | 由管理员为任务配置硬预算 | 达到任一预算后保存状态并暂停 |

“Diff 总变更行数”按新增行数与删除行数之和计算，避免通过同时新增和删除代码绕过 500 行限制。纯换行符转换、自动格式化导致的大面积无关 Diff 同样计入限制，并应由 Review Agent 标记。

管理员可以调整默认规模和预算，但调整必须发生在任务执行前，记录到审计日志，并且不能绕过人工审批、沙箱、权限和测试要求。

## 7. 任务准入规则

任务开始前必须检查：

1. 用户是否有权访问目标仓库。
2. 仓库是否为支持范围内的 GitHub Python 仓库。
3. base commit 是否存在且保持固定。
4. 是否只绑定一个 Issue，并且问题与期望行为足够明确。
5. 预计修改是否能够控制在 5 个文件和 500 行以内。
6. 是否存在可运行的 pytest 测试或可定义的复现测试。
7. 任务是否需要 V1.0 禁止的联网、宿主权限、跨仓库修改或自动部署能力。

任一关键条件不满足时，系统必须拒绝、暂停并询问用户，或将任务标记为超出范围，不得静默扩大权限与任务边界。

## 8. 任务完成标准

一个 V1.0 任务只有同时满足以下条件，才可以标记为成功：

- 根因结论引用固定 commit 上可定位的代码证据。
- 修复计划已经人工批准。
- 修改发生在任务专属临时 worktree 和受控沙箱中。
- 目标测试通过，并记录必要的回归测试结果。
- 修改没有超过已批准的文件数、行数、时间和循环预算。
- 独立 Review Agent 已检查需求覆盖、无关修改、测试质量和安全风险。
- 候选补丁已经人工批准。
- Diff、测试结果、审查结论和关键执行记录可审计、可复现。

无法复现、证据不足、预算耗尽或超出范围不是“成功”，但可以作为有明确原因和完整记录的正常终止结果。

## 9. 范围变更规则

以下变化属于产品范围变更，不能在普通开发任务中顺手加入：

- 增加新的主要编程语言或测试体系。
- 支持多仓库或多 Issue 联合修改。
- 增加自动合并或自动部署能力。
- 将单租户改为多租户。
- 弱化人工审批、沙箱、权限、审计、评测或恢复要求。

任何范围变更都必须先更新产品章程与 V1 计划，记录 ADR，并经过人工确认后才能实施。
